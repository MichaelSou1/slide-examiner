## R5 — G7 real-world prevalence (the only CRITICAL needing data)  **(P1, measure)**

**Why.** DA-C2: G7 is *defined* to be linter-blind, so "the linter is blind to G7 → we need a VLM" risks reading
as circular. The defense is showing **render-containment overflow genuinely occurs** in real generated decks.

> ⚠️ **DO NOT** borrow the pre-sales deck's "20 unfilled placeholders" as G7 evidence. Placeholders are a
> **semantic-completeness** defect (the boxes are geometrically fine — the paper itself says so); **G7 is a
> declared-legal-box / rendered-pixels-overflow geometry-render gap.** They are both linter-blind but for
> *different reasons*. Conflating them to inflate "G7 prevalence" is exactly the substitution the DA will catch
> and violates the project's citation/integrity bar. Measure **G7 specifically.**

**Steps.**
- [ ] Define a detectable G7 signature on rendered pixels: text/content bbox exceeds its declared container bbox
      while the declared box is in-margin & non-overlapping (reuse the G7 injector's own containment check, run as a
      *detector* on un-injected real renders).
- [ ] Run it over a real corpus of agent-generated decks (Zenodo10K renders and/or the deployed PowerPoint-agent
      outputs already in hand) and report the **incidence rate** of genuine render-containment overflow.
- [ ] Add one in-body sentence with the number; cite it where G7 is introduced (§3 / §5.3).

**Acceptance.** §5.3 carries a real-world incidence figure for render-containment overflow; the "constructed
class" framing is replaced by "a real, measurable IR-render gap we operationalize cleanly."
